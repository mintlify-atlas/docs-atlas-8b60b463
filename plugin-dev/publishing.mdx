---
title: "Publish Your Plugin to the Beekeeper Studio Registry"
description: "Package your plugin as a ZIP file, create a GitHub release with the required assets, and submit a pull request to the Beekeeper Studio plugin registry."
---

Once your plugin is ready, you can share it with the Beekeeper Studio community by publishing it to the official plugin registry. Published plugins appear in the in-app Plugin Manager, where users can discover, install, and update them with a single click.

<Warning>
  The plugin system is in beta and available in Beekeeper Studio 5.3+. We'd love your feedback!
</Warning>

## How the registry works

Beekeeper Studio fetches plugin metadata from a central JSON file hosted on GitHub:

```
https://raw.githubusercontent.com/beekeeper-studio/beekeeper-studio-plugins/main/plugins.json
```

Each entry in `plugins.json` points to a GitHub repository. Beekeeper Studio then reads the latest release from that repository to download your `manifest.json` and plugin ZIP file.

## Publishing process

<Steps>

<Step title="Create a public GitHub repository">

Create a public repository for your plugin. At minimum it should contain a `README.md` describing what the plugin does and how to use it.

</Step>

<Step title="Create a GitHub release">

Your release must use a **semver tag prefixed with "v"** (e.g., `v1.0.0`) and be marked as the **latest** release. It must include two specific assets:

1. `manifest.json` — your plugin manifest file
2. `{pluginId}-{version}.zip` — a ZIP archive of your built plugin files

For example, a plugin with ID `my-awesome-plugin` at version `1.0.0` requires:
- `manifest.json`
- `my-awesome-plugin-1.0.0.zip`

<Warning>
  The version in your `manifest.json` must match the git tag. If your manifest shows `"version": "1.0.0"`, your tag must be `v1.0.0`.
</Warning>

<Tabs>
  <Tab title="Using the starter template">

If your project uses the official starter template, a GitHub Actions workflow is already configured. Push a semver tag to trigger it:

```bash
git tag v1.0.0
git push origin v1.0.0
```

The workflow automatically:
- Builds your plugin
- Creates the ZIP archive
- Generates a draft GitHub release with the required assets

<Note>
  The workflow creates a **draft release**. You must go to your GitHub repository, review the draft, and click **"Publish release"** to make it public and mark it as the latest release.
</Note>

  </Tab>
  <Tab title="Manual release">

If you are not using the starter template, build your plugin and create the release manually:

1. Build your plugin output (e.g., `npm run build`)
2. Create a ZIP archive of your built files named `{pluginId}-{version}.zip`
3. Go to your GitHub repository → **Releases** → **Draft a new release**
4. Set the tag to `v{version}` (e.g., `v1.0.0`)
5. Upload both `manifest.json` and your ZIP file as release assets
6. Mark the release as the **latest release** and publish it

  </Tab>
</Tabs>

A real-world example is the [AI Shell plugin repository](https://github.com/beekeeper-studio/bks-ai-shell).

</Step>

<Step title="Submit to the plugin registry">

1. **Fork** the registry repository: [beekeeper-studio/beekeeper-studio-plugins](https://github.com/beekeeper-studio/beekeeper-studio-plugins)

2. **Edit `plugins.json`** and add your plugin entry to the array:

   ```json plugins.json
   [
     {
       "id": "my-awesome-plugin",
       "name": "My Awesome Plugin",
       "author": "Your Name",
       "description": "Brief description of what your plugin does",
       "repo": "yourusername/my-awesome-plugin"
     }
   ]
   ```

3. **Open a pull request** to the `beekeeper-studio/beekeeper-studio-plugins` repository with your change.

</Step>

<Step title="Review and approval">

After you open your pull request:

1. The Beekeeper Studio maintainers review your plugin for quality and security.
2. Once approved, your PR is merged.
3. Your plugin becomes immediately available in the Beekeeper Studio Plugin Manager.

Users can then discover, install, and receive updates for your plugin directly from within the application.

</Step>

</Steps>

## ZIP file requirements

Your ZIP archive must contain all files your plugin needs to run — HTML, CSS, JavaScript, and any other static assets. The file must be named exactly `{pluginId}-{version}.zip`.

<Tip>
  If you use a build tool like Vite, the ZIP should contain the contents of your `dist/` directory (or wherever `pluginEntryDir` points in your manifest), plus the `manifest.json` at the root. The `entry` path in your manifest views must resolve correctly relative to the ZIP root.
</Tip>

## Version compatibility

Use the `minAppVersion` field in your `manifest.json` to declare the minimum version of Beekeeper Studio required by your plugin. Beekeeper Studio will hide or disable your plugin for users running an older version.

```json manifest.json
{
  "id": "my-awesome-plugin",
  "version": "1.0.0",
  "minAppVersion": "5.4.0",
  ...
}
```

When you release a new version that uses APIs added in a newer Beekeeper Studio release, update `minAppVersion` to match. This prevents users on older versions from installing a plugin that won't work for them.

## Releasing updates

To release an update, increment your `version` field in `manifest.json`, create a new tagged release on GitHub (e.g., `v1.1.0`), and upload the new `manifest.json` and ZIP as release assets. Mark the release as the latest release. Beekeeper Studio detects the new version the next time it checks for updates, and users are notified in the Plugin Manager.

You do not need to update `plugins.json` for subsequent releases — the registry entry points to your repository, and Beekeeper Studio always fetches metadata from the latest release.
